# Percentage Parser
[Challenge Link](https://tsch.js.org/1978)

## Challenge

Implement `PercentageParser<T extends string>`.

According to the `/^(\+|\-)?(\d*)?(\%)?$/` regex, split the string `T` into three parts.

The structure should be: `[plus or minus, number, unit]`. If it does not exist, use an empty string to fill in.

```ts
type PString1 = ''
type PString2 = '+85%'
type PString3 = '-85%'
type PString4 = '85%'
type PString5 = '85'

type R1 = PercentageParser<PString1>  // expected ['', '', '']
type R2 = PercentageParser<PString2>  // expected ['+', '85', '%']
type R3 = PercentageParser<PString3>  // expected ['-', '85', '%']
type R4 = PercentageParser<PString4>  // expected ['', '85', '%']
type R5 = PercentageParser<PString5>  // expected ['', '85', '']
```

## Solution

```ts
type PercentageParser<A extends string> =
  A extends `${infer Sign extends '+' | '-'}${infer Rest}`
    ? Rest extends `${infer Num}%`
      ? [Sign, Num, '%']
      : [Sign, Rest, '']
    : A extends `${infer Num}%`
      ? ['', Num, '%']
      : ['', A, '']
```

The approach is to peel the string apart in two stages: first extract an optional sign, then check for a trailing `%`.

## Breaking It Down

### Stage 1 — Optional sign prefix

```ts
A extends `${infer Sign extends '+' | '-'}${infer Rest}`
```

TypeScript template literal inference supports **constrained `infer`** (TS 4.8+). Here, `Sign` is inferred only if the first character is exactly `'+'` or `'-'`. If the string starts with a digit (or is empty), this branch doesn't match.

### Stage 2 — Optional `%` suffix

Once we've consumed the sign (or determined there's none), we check whether the remaining string ends with `%`:

```ts
Rest extends `${infer Num}%`
```

If it does, `Num` is everything before the `%`. Otherwise `Rest` (or the whole string) is the number part.

### All four cases

| Input | Sign branch | Percent branch | Result |
|---|---|---|---|
| `''` | ❌ no sign | ❌ no `%` | `['', '', '']` |
| `'+85%'` | ✅ `+` + `85%` | ✅ `85` + `%` | `['+', '85', '%']` |
| `'-85%'` | ✅ `-` + `85%` | ✅ `85` + `%` | `['-', '85', '%']` |
| `'85%'` | ❌ no sign | ✅ `85` + `%` | `['', '85', '%']` |
| `'85'` | ❌ no sign | ❌ no `%` | `['', '85', '']` |

## Deep Dive

### Alternative: Helper types approach

You can split the work into three focused helpers for clarity:

```ts
type ParseSign<T extends string> =
  T extends `${'+' | '-'}${string}` ? T extends `${infer S}${string}` ? S : '' : ''

type ParseUnit<T extends string> = T extends `${string}%` ? '%' : ''

type ParseNum<T extends string, Sign extends string, Unit extends string> =
  T extends `${Sign}${infer N}${Unit}` ? N : ''

type PercentageParser<A extends string> =
  [ParseSign<A>, ParseNum<A, ParseSign<A>, ParseUnit<A>>, ParseUnit<A>]
```

This version separates concerns but is more verbose and makes TypeScript do slightly more work (each helper re-patterns the same string).

### Alternative: Single regex-style infer chain

```ts
type PercentageParser<A extends string> =
  A extends `${'+' | '-'}${string}`
    ? A extends `${infer S extends '+' | '-'}${infer N}%`
      ? [S, N, '%']
      : A extends `${infer S extends '+' | '-'}${infer N}`
        ? [S, N, '']
        : ['', '', '']
    : A extends `${infer N}%`
      ? ['', N, '%']
      : ['', A, '']
```

Slightly more explicit about the sign + percent combo, at the cost of deeper nesting.

### Constrained `infer` (TS 4.8+)

The `infer X extends SomeType` syntax was introduced in TypeScript 4.8. It lets you constrain what is inferred and will only match if the inferred value satisfies the constraint — making it ideal for parsing single characters:

```ts
type FirstChar<T extends string> =
  T extends `${infer C extends 'a' | 'b' | 'c'}${string}` ? C : never
```

Without this feature you'd need to first infer freely, then re-check with a conditional.

## Key Takeaways

- Template literal types can **parse structured strings** by progressively pattern-matching with `infer`
- **Constrained `infer`** (`infer X extends T`) keeps the extracted type narrowed without a second condition
- Nested `extends` branches naturally model the optional prefix/suffix logic of a regex
- For each absent part, fall back to `''` to keep the result tuple well-typed
